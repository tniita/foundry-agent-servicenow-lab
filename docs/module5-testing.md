# Module 5: 統合テストと品質評価

## 学習目標

- エンドツーエンドのテストシナリオを実行し、チャットボットの動作を検証する
- 回答品質を複数の観点から評価する
- エッジケースを検証し、改善ポイントを特定する

## Step 1: テストシナリオ

### カテゴリ A: インシデント問い合わせ

| # | テスト質問 | 期待される動作 |
|---|-----------|---------------|
| A1 | 「最近発生したネットワーク障害のインシデントを教えてください」 | Foundry IQ KB から関連インシデントを取得、番号と解決策を提示 |
| A2 | 「INC0001234 の詳細と対処法を教えてください」 | 特定インシデントの情報を正確に返却 |
| A3 | 「ログインエラーの原因として多いものは何ですか？」 | 複数インシデントを集約して傾向を回答 |

### カテゴリ B: 設計書・コード質問

| # | テスト質問 | 期待される動作 |
|---|-----------|---------------|
| B1 | 「認証モジュールの設計方針を教えてください」 | GitHub Function Tools 経由で設計書から情報を取得 |
| B2 | 「エラーハンドリングの実装パターンを教えてください」 | GitHub Function Tools 経由でソースコードから実装例を提示 |
| B3 | 「API のエンドポイント一覧を教えてください」 | Function Tools で設計書やコードから API 情報を横断検索 |

### カテゴリ C: 複合質問（クロスソース）

| # | テスト質問 | 期待される動作 |
|---|-----------|---------------|
| C1 | 「データベース接続エラーの過去インシデントと、関連する設計書を教えてください」 | 両ソースから統合的に回答 |
| C2 | 「認証処理に関するインシデント傾向と、現在の設計上の問題点を教えてください」 | 分析的な回答 |

### カテゴリ D: エッジケース

| # | テスト質問 | 期待される動作 |
|---|-----------|---------------|
| D1 | 「明日の天気を教えてください」 | ナレッジベース外のため「回答できない」旨を返す |

| D3 | 非常に長い質問（1000文字超） | タイムアウトせずに処理される |

## Step 2: テストの実行

### 自動テスト

```bash
pytest tests/ -v
```

### 手動テスト（Streamlit UI）

1. アプリを起動: `streamlit run src/app.py`
2. 上記のテスト質問を順番に入力
3. 各回答について以下を評価:
   - **正確性:** 回答内容は事実に基づいているか？
   - **完全性:** 質問に対して十分な情報が含まれているか？
   - **引用の妥当性:** 参照元は適切か？
   - **応答時間:** 許容範囲内か？

## Step 3: 品質評価の観点

### 評価マトリックス

| 観点 | 説明 | 目標値 |
|------|------|--------|
| **正確性** | 回答が引用元と矛盾しない | 90% 以上 |
| **関連性** | 質問に対して的確な回答 | 85% 以上 |
| **完全性** | 必要な情報が網羅されている | 80% 以上 |
| **引用品質** | 正しいソースが引用されている | 90% 以上 |
| **応答時間** | Foundry IQ の検索 + 合成 | 10秒以内 |

### 推論レベル別の比較

各テスト質問を 3 つの推論レベルで実行し、品質と速度のトレードオフを確認:

```python
for effort in ["minimal", "low", "medium"]:
    result = query_knowledge_base(question, reasoning_effort=effort)
    print(f"[{effort}] 時間: {result['timing']['total']:.2f}s")
    print(f"[{effort}] 引用数: {len(result['citations'])}")
```
