## 1. システム目的

社内の主要業務プロセス（勤怠管理・経費精算・社内通知）を統合し、従業員の業務効率化と管理部門の運用負荷軽減を実現する。

### 1.1 背景

- 既存の勤怠管理は Excel ベースで月末に手動集計 → ヒューマンエラー多発
- 経費精算は紙の申請書 → 承認に平均5営業日
- 社内通知はメール一斉配信 → 既読率 30% 以下

### 1.2 目標

| 指標 | 現状 | 目標 |
|------|------|------|
| 勤怠集計所要時間 | 3日/月 | 自動化（0日） |
| 経費精算承認リードタイム | 5営業日 | 1営業日 |
| 社内通知既読率 | 30% | 80%以上 |
| システム稼働率 | — | 99.9% (SLA) |

## 3. 非機能要件

### 3.1 性能要件

| 項目 | 要件 |
|------|------|
| 同時接続数 | 最大 500 ユーザー |
| API レスポンス | 95 パーセンタイルで 500ms 以内 |
| 月次バッチ処理 | 1,000 名分の勤怠集計を 10 分以内 |
| ファイルアップロード | 最大 10MB / 1ファイル |

### 3.2 可用性要件

| 項目 | 要件 |
|------|------|
| 稼働率 | 99.9%（月間ダウンタイム 43 分以内） |
| RTO | 1 時間 |
| RPO | 5 分 |
| メンテナンスウィンドウ | 毎月第2日曜 02:00-06:00 JST |

### 3.3 セキュリティ要件

- Azure AD による SSO / MFA 必須
- 通信は全て TLS 1.3
- 個人情報はデータベースレベルで暗号化 (TDE)
- 操作ログは Azure Monitor に 1 年間保存
- CORS ポリシーにより許可ドメインのみアクセス可

## 5. 運用設計

### 5.1 監視項目

| 監視対象 | ツール | アラート条件 |
|---------|--------|-------------|
| API レスポンスタイム | Application Insights | P95 > 1000ms |
| エラー率 | Application Insights | 5xx > 1% / 5分 |
| DB コネクションプール | Azure Monitor | 使用率 > 80% |
| ディスク使用率 | Azure Monitor | > 85% |
| SSL 証明書有効期限 | Azure Monitor | 残り 30 日以下 |

### 5.2 バックアップ

| 対象 | 方式 | 保持期間 |
|------|------|---------|
| PostgreSQL | Azure 自動バックアップ (PITR) | 35 日 |
| Blob Storage | GRS (地理冗長) | 無期限 |
| アプリケーションログ | Log Analytics | 90 日 |

### 5.3 障害対応フロー

```
障害検知 → アラート発報 → 運用チーム確認 → 影響範囲特定
    → 一次対応（再起動/切り戻し）→ 恒久対応 → ポストモーテム
```

障害レベル定義:
- **P1 (Critical)**: 全ユーザーがシステム利用不可 → 30分以内に対応開始
- **P2 (High)**: 主要機能が利用不可 → 1時間以内に対応開始
- **P3 (Medium)**: 一部機能に影響 → 4時間以内に対応開始
- **P4 (Low)**: 軽微な問題 → 翌営業日対応
